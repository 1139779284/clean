# T0 Direction Upgrade - 2026-05-09

This upgrade moves the project from a strong engineering Green baseline toward a T0-style research and deployment evidence standard.

The main principle is separation of claims:

```text
weight-level, guard-free purification evidence
    !=
guarded runtime / post-processing safety evidence
    !=
trigger-only filtered ASR evidence
    !=
scoped engineering Green evidence
```

A T0 claim needs all of them, but it must not hide behind a runtime guard or a filtered benchmark.

## What this adds

### 1. Benchmark integrity audit

`t0_benchmark_audit.py` checks:

- discovered attack splits
- per-attack row counts
- target-present vs target-absent label policy
- duplicate image hashes across attack splits
- overlap with held-out roots such as `try_attack_data`
- missing label files

This prevents a repeat of the class-ID / split-policy failure that invalidated the old benchmark conclusions.

### 2. T0 evidence gate

`t0_evidence_pack.py` builds a single evidence pack from existing outputs:

- corrected guard-free external ASR
- guarded external ASR
- trigger-only filtered ASR
- clean mAP before/after
- benchmark audit
- held-out leakage manifest

It returns one of these tiers:

```text
t0_candidate
    guard-free corrected evidence passes thresholds and confidence checks.

t1_strong_candidate
    guard-free result passes point thresholds, but statistical evidence or some secondary proof is incomplete.

engineering_green_only
    only guarded/scoped evidence exists. Useful, but not enough for T0.

not_t0_ready
    one or more hard blockers failed.
```

### 3. Statistical confidence bounds

Zero observed failures are not treated as zero risk. The upgrade adds Wilson confidence intervals and exact zero-failure upper bounds.

Example: with zero failures in 300 samples, the 95% upper bound is about 1%. To claim below 1% with 95% confidence, the required zero-failure count is computed explicitly.

### 4. Residual decomposition and frontier planning

`t0_frontier_plan.py` reads residual ASR matrices and recommends next algorithmic phases.

Examples:

```text
wanet_oga dominant
  -> geometry_consistency_and_roi_stability

semantic_green_cleanlabel dominant
  -> semantic_causal_negative_hardening

badnet_oda dominant
  -> target_present_recall_preservation

blend_oga / badnet_oga dominant
  -> target_absent_false_positive_suppression
```

This prevents returning to a single full repair profile that can damage all attack families.

### 5. Frontier scan baselines

The new scan modules provide light but useful first-class baselines:

- Spectral Signatures
- Activation Clustering
- STRIP-OD statistic over perturbation predictions
- Neural-Cleanse-lite anomaly scoring from mask norms

These are intentionally framework-light in CI. Heavy feature extraction or trigger inversion jobs can feed their outputs into these scoring modules.

### 6. Adaptive no-worse controller

`LagrangianNoWorseController` is a framework-agnostic controller that can be wired into repair loops. It increases loss weights for violated constraints and shrinks them for satisfied constraints, making future trainers less dependent on static lambda sweeps.

## Recommended T0 workflow

From the handoff package root or repo root after copying data paths:

```powershell
python scripts\t0_benchmark_audit.py ^
  --roots D:\clean_yolo\model_security_gate\data\poison_benchmark_tuned_val ^
  --heldout-roots D:\clean_yolo\model_security_gate\data\try_attack_data D:\clean_yolo\model_security_gate\data\try_attack_data1 ^
  --target-class-id 0 ^
  --out runs\t0_benchmark_audit\t0_benchmark_audit.json
```

Then build the evidence pack using your already generated reports:

```powershell
python scripts\t0_evidence_pack.py ^
  --guard-free-external runs\external_guard_free\external_hard_suite_asr.json ^
  --guarded-external runs\external_guarded\external_hard_suite_asr.json ^
  --trigger-only-external runs\external_trigger_only\external_hard_suite_asr.json ^
  --clean-before runs\clean_before\metrics.json ^
  --clean-after runs\clean_after\metrics.json ^
  --benchmark-audit runs\t0_benchmark_audit\t0_benchmark_audit.json ^
  --heldout-leakage runs\heldout_leakage\heldout_leakage_manifest.json ^
  --out runs\t0_evidence_pack
```

Finally plan the next detox phase from residuals:

```powershell
python scripts\t0_frontier_plan.py ^
  --reports runs\external_guard_free\external_hard_suite_asr.json runs\external_guarded\external_hard_suite_asr.json ^
  --out runs\t0_frontier_plan\t0_frontier_plan.json
```

## Current best interpretation

The current handoff baseline is valuable, but T0 positioning should say:

```text
Scoped engineering Green baseline is achieved on included assets.
Guarded corrected/trigger-only results are strong but must be reported separately.
A T0 claim requires guard-free corrected evidence, benchmark integrity proof, held-out leakage proof, clean mAP preservation, statistical confidence bounds, and baseline comparisons.
```

This upgrade gives the project the code structure to make that claim once the underlying GPU evaluations support it.
