## T0 OD Defense Leaderboard (new, contribution 3 scaffold)

A faithful, BackdoorBench-style OD defense comparison layer is now
implemented:

- Code: `model_security_gate/t0/defense_leaderboard.py`
- CLI: `scripts/t0_defense_leaderboard.py` (pixi task `t0-defense-leaderboard`)
- Tests: `tests/test_t0_defense_leaderboard.py`
- Example manifest: `configs/t0_defense_leaderboard.example.yaml`

It adapts Wu et al., 2022 ([BackdoorBench](https://arxiv.org/abs/2206.12654))
DER to object detection and tightens it in three directions the source
benchmark does not currently enforce:

1. **Per-attack paired McNemar test.** Given the per-image rows in external
   hard-suite reports, every attack receives an exact two-sided binomial
   McNemar p-value. A defense that lowers mean ASR but fails McNemar
   significance on the target attack is surfaced explicitly.
2. **No-regression dominance.** A defense is rejected if any attack family
   regresses relative to its poisoned baseline. BackdoorBench's single DER
   number can hide this.
3. **OD-specific clean term.** Clean accuracy is replaced by mAP50-95; the
   clean-mAP drop is bounded by the project's existing 3-percentage-point
   acceptance tolerance.

Ranking primary key: `accepted`, then OD-DER (desc), then max defended ASR
(asc), then mAP drop (asc).

Next step: feed NAD / ANP / RNP / I-BAU / Neural-Cleanse-lite / STRIP-OD
runs into this leaderboard once each baseline produces paired
`external_hard_suite_asr.json` reports on the corrected suite. The manifest
schema in `configs/t0_defense_leaderboard.example.yaml` is the drop-in
location.

## T0 OD Defense Certificate (CFRC, contribution 3 final form)

**Positioning.** CFRC is the evidence layer for the detox main line; it is
neither a replacement nor a competitor. The project structure is:

- contribution 1: OD multi-attack detox algorithm (`detox/`, `guard/`);
- contribution 2: attack zoo + poisoned-model matrix
  (`attack_zoo/`, `plan_t0_poison_model_matrix.py`);
- contribution 3: CFRC certification protocol (this section).

CFRC does **not** produce a detox model. It takes a (poisoned baseline,
defended model) pair from the main line and produces statistical evidence
strong enough for a T0-style claim.

**Implementation.**

- Code: `model_security_gate/t0/defense_certificate.py`
- CLI: `scripts/t0_defense_certificate.py` (pixi task `t0-defense-certificate`)
- Tests: `tests/test_t0_defense_certificate.py`
- Shared manifest schema with the leaderboard: `configs/t0_defense_leaderboard.example.yaml`

**Method.** CFRC combines three corrections that the BackdoorBench single
DER number does not apply:

1. **Paired bootstrap CI on ASR reduction.** Per-image paired resampling of
   `r_i = success_i^poisoned - success_i^defended` produces the two-sided
   bootstrap interval and the one-sided lower bound used by CMR. Seeded and
   deterministic (`DefenseCertificateConfig.seed`).
2. **Holm-Bonferroni family-wise error correction.** Raw per-attack McNemar
   p-values are sorted and adjusted with the step-down Holm procedure
   (Holm, 1979). A defense is significant on an attack iff `adj_p <= alpha`
   AND the discordant-pair direction favors the defense.
3. **Certified Minimum Reduction (CMR).** `cmr_asr = min over attacks of
   bootstrap_lower_bound(ASR_reduction)`. A defense is certified iff every
   tracked attack reduces ASR by at least `min_certified_reduction` at the
   joint lower confidence level, clean mAP50-95 drop stays within tolerance,
   and no attack regresses.

**Target claim it enables.** In the CFRC report, the headline sentence is
no longer *"mean ASR dropped 60%"*. It becomes:

> Across 13 tracked attacks, the method reduces ASR by at least x with 95%
> confidence on every attack family (Holm-Bonferroni corrected), while
> preserving clean mAP50-95 within the project tolerance.

This is what the main-line detox matrix will ultimately be graded on.

## Contribution 1 — Lagrangian multi-attack wiring (new, main-line algo)

The project already shipped `MultiAttackLagrangianController` and
`default_t0_constraints`, but they lived outside the training loop.  This
made Hybrid-PURIFY's per-phase lambdas static dataclass constants that could
only be moved by the binary `aggressive_mode` toggle.  The controller is now
wired into the cycle loop (opt-in, default off for full backward
compatibility):

- `HybridPurifyConfig.use_lagrangian_controller` + `lagrangian_lambda_lr`,
  `lagrangian_lambda_min`, `lagrangian_lambda_max`, `lagrangian_decay`,
  `lagrangian_base_scale`, `lagrangian_max_scale`, `lagrangian_min_scale`.
- `scripts/hybrid_purify_detox_yolo.py` exposes the same knobs under
  `--use-lagrangian-controller --lagrangian-*` for CLI and YAML overrides.
- Helpers `_build_lagrangian_controller`, `_normalize_metric_keys`,
  `_bucket_for_attack`, `_bucket_scales_from_lambdas`,
  `_apply_lagrangian_weights` (all pure Python, tested).
- `_run_feature_purifier_phase` accepts `lagrangian_lambdas` and multiplies
  every phase's bucket-relevant `lambda_*` by the current per-attack scale
  before calling `run_strong_detox_training`.
- End-of-cycle: the external per-attack ASR (plus `clean_map_drop`) is
  pushed into `controller.update(...)` and the full trace (per-cycle
  metrics, lambdas, violations) is persisted to
  `hybrid_purify_manifest.json` under `lagrangian_controller.trace`.
- Backward compatibility: `use_lagrangian_controller=False` (default) keeps
  bit-for-bit identical behaviour; existing manifests and acceptance gates
  do not change.

Tests: `tests/test_hybrid_purify_lagrangian.py` (15 cases) cover
normalization, bucket mapping, scale clamping, invariance under empty
lambdas, controller construction, and the dynamics invariant that violated
attacks end with larger lambdas than satisfied ones at the same starting
point.

Algorithmic claim this unlocks:

> We reformulate multi-attack OD detox as an outer Lagrangian on per-attack
> ASR and clean mAP, with each cycle adapting per-phase loss weights based
> on observed external-suite violations.  The weights are bucketed per
> attack family (oga / oda / wanet / semantic / clean) to remain
> interpretable.  With CFRC certification, we report whether adaptive
> lambda dominates static lambda at the CMR level across the matrix.

This is the Contribution-1 upgrade.  Contributions 2 and 3 remain
unchanged; CFRC now has a strictly stronger candidate to certify.

## T0 Detox Ablation Planner (new connector between contributions 1 and 3)

Wiring the adaptive controller into Hybrid-PURIFY created a natural
ablation: same poisoned baseline, same external hard suite, identical
training budget, only ``use_lagrangian_controller`` flipped.  The planner
formalizes that experiment end-to-end:

- Code: `model_security_gate/t0/ablation_plan.py`
- CLI: `scripts/t0_detox_ablation_plan.py` (pixi task `t0-detox-ablation-plan`)
- Tests: `tests/test_t0_ablation_plan.py` (13 cases)
- Example config: `configs/t0_detox_ablation.example.yaml`
- Concrete local plan: `configs/t0_detox_ablation_local.yaml`
- Current plan artifact (pending GPU runs):
  `runs/t0_detox_ablation_local_2026-05-10/T0_DETOX_ABLATION_RUNBOOK.md`

What the planner does:

1. Reads a YAML spec describing the poisoned baseline and data paths.
2. Emits the exact Hybrid-PURIFY-OD CLI command per arm, including the
   ``--use-lagrangian-controller`` flag where appropriate.
3. Probes disk for already-completed runs (`hybrid_purify_manifest.json`,
   `external_hard_suite_asr.json`, clean metrics).
4. For each completed run, extracts paths from the hybrid manifest and
   adds a `DefenseEntry`-shaped row to `cfrc_manifest.json`.  Pending
   arms are recorded under `skipped` so the final table is reproducible.
5. Writes a human-readable runbook that doubles as the GPU operator's
   checklist.

Once every arm is done, CFRC is one command away:

```powershell
pixi run t0-defense-certificate `
  --manifest runs/t0_detox_ablation_local_2026-05-10/cfrc_manifest.json `
  --out runs/t0_detox_ablation_local_2026-05-10/cfrc
```

This closes the contribution-1 to contribution-3 loop: no manual JSON
editing between Hybrid-PURIFY finishing and the CFRC certificate being
emitted.

### Smoke-first run order (mandatory before full)

A smoke-scale spec mirrors the sizing of
`runs/hybrid_purify_corrected_suite_smoke_2026-05-09`: `imgsz=416`,
`cycles=1`, one epoch per stage, `max_images=800`, 40 eval images per
attack.  Each arm fits in ~30-45 min on a single RTX 4060 Laptop.

- Smoke spec: `configs/t0_detox_ablation_smoke.yaml`
- Smoke runbook: `runs/t0_detox_ablation_smoke_2026-05-10/T0_DETOX_ABLATION_RUNBOOK.md`
- Smoke CFRC manifest (pending): `runs/t0_detox_ablation_smoke_2026-05-10/cfrc_manifest.json`

The smoke run is **not a publication result**.  Its purpose is plumbing
validation only:

1. both arms finish without crashing;
2. Hybrid-PURIFY writes manifests with `external_json` and
   `clean_metrics`;
3. `t0-detox-ablation-plan` second run finds them and fills the CFRC
   manifest `entries` list automatically;
4. `t0-defense-certificate` emits the certificate end-to-end.

Wide CFRC confidence intervals are expected on smoke sizes (n=40 per
attack) and do not indicate a weak defense.  Only after smoke validation
passes do we run `configs/t0_detox_ablation_local.yaml` at the full
sizing for the publication-grade certificate.
